# 调研报告：好奇 Agent 原型研究

> **项目代号**: Curious Agent  
> **版本**: v0.1 MVP  
> **日期**: 2026-03-19  
> **目的**: 为 OpenCode 提供可落地的 Agent 自主意识架构参考

---

## 一、研究背景与目标

### 1.1 问题陈述

当前主流 Agent 框架（如 LangChain Agents、AutoGPT、AutoGen）存在三个根本性缺陷：

1. **被动响应** — Agent 只在用户明确提问时工作，无法自主发现知识缺口
2. **缺乏元认知** — 没有持续监控"我知道什么、不知道什么"的机制
3. **短视探索** — 探索行为碎片化，无法形成积累性的知识体系

### 1.2 研究目标

设计并实现一个**好奇驱动的 Agent 原型**：
- 主动识别知识缺口
- 持续、积累性地探索未知领域
- 基于好奇心优先级动态调度探索任务
- 将发现转化为可复用的知识资产

### 1.3 核心创新点

| 维度 | 传统 Agent | Curious Agent |
|------|-----------|---------------|
| 触发方式 | 用户指令 | 好奇心驱动 |
| 知识管理 | 无状态 / 会话级 | 持久化知识图谱 |
| 自我监控 | 无 | 元认知监控层 |
| 探索策略 | 随机 / 手动 | 动态评分 + 优先级队列 |
| 遗忘机制 | 无 | 基于时间的衰减 |

## 二、文献调研

### 2.1 元认知 (Metacognition) 理论基础

**定义** (Flavell, 1979)：元认知是"关于认知的认知"，包含三个核心组件：

```
┌─────────────────────────────────────────┐
│            元认知架构                      │
├─────────────────────────────────────────┤
│  元认知知识 ←→ 元认知监控 ←→ 元认知调节     │
│  (我知道什么)  (我在想什么)  (我该调整了)  │
└─────────────────────────────────────────┘
```

**Agent 映射**：
- 元认知知识 → 知识图谱中的已知/未知状态
- 元认知监控 → 好奇心引擎的持续评分机制
- 元认知调节 → 探索器动态选择探索策略

### 2.2 相关框架调研

#### Generative Agents (Stanford, Park et al., 2023)

**核心架构**：
- **记忆流 (Memory Stream)** — 将经验压缩为 reflection，分层检索
- **反应 (ReAct-style)** — 观察 → 推理 → 行动
- **计划 (Planning)** — 自然语言计划 + 动态重规划

**局限性**：
- 非真正元认知（依赖 LLM 被动推理）
- O(N²) 记忆检索，可扩展性差
- 缺乏持续后台监控

#### Reflexion (NeurIPS 2023)

**核心机制**：
- 用语言反馈替代梯度更新
- Agent 执行 → 自我评估 → verbal reflection → 下一轮
- **创新点**：将失败经验转化为未来决策依据

**局限性**：
- 仍是事后反思，非主动预防
- 依赖外部评估信号

#### Autono (arXiv:2504.04650, 2025)

**核心机制**：
- 基于 ReAct 的高度鲁棒自主 Agent 框架
- 增强了错误处理和恢复机制

**局限性**：
- 仍是单 Agent 架构
- 无内在好奇心机制

#### RE-TRAC (arXiv:2602.02486, 2026)

**核心机制**：
- 递归轨迹压缩，用于深度搜索 Agent
- 减少长上下文 Token 消耗

**局限性**：
- 专注于压缩，非探索驱动

### 2.3 关键技术比较

| 框架 | 元认知 | 好奇心驱动 | 持久化 | 评分机制 | 开源 |
|------|--------|-----------|--------|---------|------|
| Generative Agents | ❌ | ❌ | 内存级 | 无 | ✅ |
| Reflexion | 部分 | ❌ | 无 | 无 | ✅ |
| Autono | ❌ | ❌ | 无 | 无 | ✅ |
| RE-TRAC | ❌ | ❌ | 无 | 无 | ✅ |
| **Curious Agent** | ✅ | ✅ | ✅ | ✅ | ✅ |

## 三、技术选型

### 3.1 为什么选择 Python

- Agent 领域主流语言，工具链成熟
- 丰富的 LLM/Embedding 库（LangChain, LlamaIndex）
- 快速原型迭代

### 3.2 为什么选择 JSON 持久化（而非数据库）

MVP 阶段目标：
- **简单性** — 无需额外部署
- **可追溯性** — JSON 可直接查看和调试
- **够用性** — 小规模知识图谱（<1000节点）JSON 性能足够

**后续迭代**建议：
- SQLite → 轻量，无需服务
- Neo4j → 图数据库，适合复杂关系查询

### 3.3 搜索 API 选择

| API | 优点 | 缺点 |
|-----|------|------|
| Google Custom Search | 质量高 | 需要配置，限额 |
| Serper | 质量高 | 需要 Key |
| Bocha | 中文优化，已配置 | 国际内容有限 |
| DuckDuckGo | 免费 | 质量一般 |

**当前使用**：Bocha (`/v1/web-search`)

## 五、局限性说明

1. **搜索质量依赖 API** — 当前 Bocha 对国际前沿论文覆盖有限
2. **评分算法主观** — 启发式参数需大量实验调优
3. **无真正理解** — 基于关键词匹配，非语义理解
4. **规模限制** — JSON 持久化无法支撑大规模知识图谱
5. **无多模态** — 仅处理文本，无图像/音频探索
