# Curious Agent v0.2 设计方案

> **版本**: v0.2 | **状态**: 待实现 | **设计时间**: 2026-03-19  
> **目标**: 主动触发 + 分层探索

---

## 一、设计概述

### 1.1 背景

v0.1 实现了基础的"好奇心驱动探索"能力，但存在两个核心局限：

1. **被动触发** - 纯 cron 驱动（每30分钟一次），缺乏时间感和事件响应
2. **浅层探索** - 仅一层 web search，无法深入论文获取洞察

### 1.2 设计目标

| 改进 | 目标 | 验收标准 |
|------|------|---------|
| **主动触发** | 建立时间感知 + 事件驱动体系 | 支持定时/自动/对话三种触发方式 |
| **分层探索** | 实现三档深度探索机制 | shallow(<30s) / medium(3-5min) / deep(10-15min) |

## 二、改进一：主动触发体系

### 2.1 触发信号体系

```
触发信号
  ├── ⏰ 定时节律（被动基准）
  │     ├── 早安探索（09:00）--depth shallow
  │     └── 晚安探索（21:00）--depth deep
  │
  ├── 🔗 知识缺口触发（主动核心）
  │     ├── 探索结果发现新关键词 → 自动入队
  │     │   source="auto:found in {parent_topic}"
  │     └── 图谱中某话题孤立 → 主动填补关联
  │
  └── 💬 对话触发（用户交互）
        └── POST /api/curious/trigger → 后台执行
```

### 2.2 触发 API 设计

**POST /api/curious/trigger** - 同步投递，后台执行

```bash
curl -X POST http://10.1.0.13:4849/api/curious/trigger \
  -H "Content-Type: application/json" \
  -d '{
    "topic": "knowledge graph embedding",
    "depth": "medium",
    "priority": "high"
  }'
```

**响应**: 
```json
{
  "status": "accepted",
  "topic": "knowledge graph embedding",
  "depth": "medium",
  "estimated_time": "3-5分钟",
  "message": "探索已启动，完成后将通过现有机制通知"
```

**实现机制**:
- HTTP 立即返回（无超时风险）
- 后台 threading 启动探索
- 完成后通过飞书/日志通知（复用现有机制）

## 三、改进二：分层探索机制

### 3.1 三层探索架构

```
探索深度
  │
  ├─ Layer 1 ── 快（~30秒）
  │     └─ Web Search (Bocha API)
  │          → 快速概览 + 发现 arXiv 链接
  │
  ├─ Layer 2 ── 中（~3-5分钟）
  │     ├─ Layer 1 结果
  │     └─ arXiv 深度分析
  │          → arXiv API 获取摘要
  │          → 相关性评分判断
  │          → 有价值则下载 PDF 全文解析
  │          → 提取：方法、贡献、局限
  │
  └─ Layer 3 ── 深（~10-15分钟）
        ├─ Layer 1+2 结果（2-3篇论文）
        └─ LLM 洞察生成 (minimax-m2.7)
             → 多论文对比分析
             → 跨论文趋势观察
             → 生成对比表 + 洞察
             → 判断是否通知用户
```

### 3.2 各层详细设计

#### Layer 1 - Web Search（现有能力）

**输入**: topic, depth="shallow"  
**输出**: 搜索摘要 + arXiv 链接列表

**保持现状**，但增强输出：
```json
{
  "layer": 1,
  "findings": "搜索摘要...",
  "arxiv_links": ["https://arxiv.org/abs/2401.02009", ...],
  "duration": 25
}
```

#### Layer 2 - arXiv 深度分析

**输入**: topic, arxiv_links  
**输出**: 论文分析报告

**流程**:
```python
class ArxivAnalyzer:
    def analyze(self, topic: str, arxiv_links: list) -> dict:
        results = []
        for link in arxiv_links[:3]:  # 最多分析3篇
            # 1. 调用 arXiv API 获取元数据
            paper = self.fetch_arxiv_metadata(link)
            
            # 2. 计算与 topic 的相关性评分
            relevance_score = self.compute_relevance(topic, paper)
            
            if relevance_score > 0.6:  # 阈值可配置
                # 3. 下载 PDF 全文
                pdf_path = self.download_pdf(link)
                
                # 4. 解析 PDF（前2页）
                full_text = self.extract_pdf_text(pdf_path, pages=2)
                
                # 5. 深度分析
                analysis = {
                    "title": paper.title,
                    "authors": paper.authors,
                    "abstract": paper.abstract,
                    "relevance_score": relevance_score,
                    "full_text_preview": full_text[:2000],
                    "key_findings": self.extract_key_findings(full_text)
                }
                results.append(analysis)
        
        return {
            "layer": 2,
            "papers_analyzed": len(results),
            "analyses": results,
            "duration": elapsed_time
        }
```

**相关性评分算法**:
```python
def compute_relevance(topic: str, paper) -> float:
    # 基于 TF-IDF 或简单词频
    topic_words = set(topic.lower().split())
    
    title_score = len(topic_words & set(paper.title.lower().split())) / len(topic_words)
    abstract_score = len(topic_words & set(paper.abstract.lower().split())) / len(topic_words)
    
    # 加权：标题匹配权重更高
    return title_score * 0.6 + abstract_score * 0.4
```

#### Layer 3 - LLM 洞察生成

**输入**: 2-3篇论文的 Layer 2 分析结果  
**输出**: 对比洞察报告  
**LLM**: minimax-m2.7

**Prompt 模板**:
```
你是一位 AI 研究分析专家。请基于以下论文分析，生成深度洞察报告。

研究主题: {topic}

论文1:
标题: {paper1.title}
作者: {paper1.authors}
摘要: {paper1.abstract}
关键发现: {paper1.key_findings}

论文2:
...

论文3:
...

请提供以下分析:
1. 方法论对比表（维度：方法类型、创新点、局限性）
2. 核心贡献总结
3. 跨论文趋势观察
4. 对该研究领域的建议
5. 是否值得深入探索（是/否 + 理由）
```

**输出格式**:
```json
{
  "layer": 3,
  "comparison_table": {...},
  "key_contributions": "...",
  "trend_observation": "...",
  "recommendation": "...",
  "worth_deep_dive": true,
  "insight_score": 8.5,
  "duration": 420
}
```

### 3.3 探索深度参数

## 八、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| arXiv API 限流 | Layer 2 失败 | 实现指数退避重试，失败时降级到仅摘要 |
| PDF 下载超时 | Layer 2 卡住 | 设置 30s 超时，大文件跳过 |
| minimax API 慢 | Layer 3 超时 | 设置 60s 超时，失败时返回"洞察生成失败" |
| 磁盘空间不足 | PDF 积累 | 定期清理 /tmp，保留最近 10 个 PDF |
| 重复自动入队 | 队列膨胀 | 去重检查，相同 topic 30 天内不再入队 |

## 九、设计决策记录

| 决策 | 选项 | 选择 | 理由 |
|------|------|------|------|
| 触发 API | A.异步+状态查询 B.同步+后台执行 | **B** | 更简单，复用现有通知机制 |
| PDF 获取 | A.直接下载 B.arXiv API优先 C.混合 | **C** | 优先 API，有价值再下载全文 |
| 内容分析 | A.关键词匹配 B.相关性评分 C.启发式 | **B** | 基于 topic 相关性更精准 |
| LLM 选择 | OpenAI/本地/minimax | **minimax-m2.7** | 用户指定 |
| 任务存储 | state.json/jobs.json/SQLite | **内存+通知** | 不需要持久化任务状态 |
