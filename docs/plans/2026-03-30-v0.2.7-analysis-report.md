### 🔴 问题 1: 被动式补丁策略

**症状**:
- 5 个独立根因 (G1-G5) 各自孤立修复
- 缺乏系统性的架构审视
- Bug 链描述清晰但解决方案碎片化

**深层问题**:
```
当前思维: "发现 Bug → 定位代码 → 打补丁"
应有思维: "为什么这些 Bug 能进入生产？系统设计哪里出了问题？"
```

**根因**:
1. 缺乏统一的数据流设计
2. 状态管理分散在多个模块
3. 没有数据一致性约束

---

### 🔴 问题 2: 数据流设计缺陷

**核心问题**: Parent Link 的传递机制混乱

当前设计的问题:
```
api_inject(topic, parent)
  ↓ 只调 add_curiosity()
  ↓ parent 信息丢失在队列中
  ↓ 探索完成后试图从队列"反向推断"parent
  ↓ 推断逻辑错误 (Bug 3+4)
  ↓ parent 关系断裂
```

**v0.2.7 的修复局限**:
- 在入口处写入 KG，但队列仍不携带 parent 信息
- 这只是绕过了问题，没有解决根本矛盾
- 队列和 KG 的数据模型不一致问题仍然存在

**建议的根本解决**:
```python
# 队列项应携带完整的血缘信息
{
    "topic": "xxx",
    "lineage": {
        "parent_topic": "Agent Harness",
        "injected_by": "r1d3",
        "decomposition_depth": 1
    }
}
```

### 🔴 问题 3: 状态管理脆弱性

**症状**:
- `curiosity_queue` 状态与 KG 状态不同步
- SpiderAgent/DreamAgent/Async 三条路径各自管理状态
- 没有统一的状态机

**具体问题对比**:

| 路径 | 状态管理 | 问题 |
|------|---------|------|
| SpiderAgent | DreamInbox → 不经过队列 | 队列状态与实际探索脱节 |
| Async | 直接触发 `_explore_in_thread` | 状态从 `pending` 直接跳到 `done` |
| Claim | Phase 0 新增 | 需要额外的 `claim_pending_item()` |

**建议**: 采用统一的状态机

```python
class ExplorationStateMachine:
    STATES = {
        'pending': {'to': ['claimed', 'cancelled']},
        'claimed': {'to': ['exploring', 'released']},
        'exploring': {'to': ['done', 'paused', 'failed']},
        # ...
    }
```

### 🟡 问题 4: 质量评估架构缺陷

**QualityV2 的问题**:
```python
quality = assess_quality(
    topic=topic,
    findings=findings_dict,
    knowledge_graph=None  # ← 设计问题
)
```

**根本问题**: QualityV2 的设计假设调用方会传入 KG，但调用方不知道应该传什么。

**建议**: 懒加载 + 缓存模式
```python
class QualityV2Assessor:
    def __init__(self):
        self._kg_module = None
    
    @property
    def kg(self):
        if self._kg_module is None:
            from core import knowledge_graph as kg_module
            self._kg_module = kg_module
        return self._kg_module
```

## 三、解决方案

### 方案 1: 统一状态机设计 (替代 Phase 0+3)

```python
class ExplorationStateMachine:
    """统一的状态管理，替代碎片化的状态更新"""
    
    STATES = {
        'pending': {'to': ['claimed', 'cancelled'], 'actor': 'system'},
        'claimed': {'to': ['exploring', 'released'], 'actor': 'agent'},
        'exploring': {'to': ['done', 'paused', 'failed'], 'actor': 'agent'},
        'done': {'to': [], 'actor': 'system'},
        'paused': {'to': ['pending'], 'actor': 'system'},
        'failed': {'to': ['pending'], 'actor': 'system'},
    }
```

**优势**:
- 所有路径统一走状态机
- 状态转换可追溯
- 非法转换可拦截
