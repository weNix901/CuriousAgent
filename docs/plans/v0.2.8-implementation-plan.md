# v0.2.8 知识断言评估实施计划

> **目标**: 解决 QualityV2 误判问题，通过知识断言生成和相似度检测准确评估探索质量

**核心问题**: 当前 QualityV2 依赖 LLM 判断 summary 是否是 topic 改写，容易误判（假 0.0）

**解决方案**: 从 findings 生成 3 个具体知识断言，通过 embedding 相似度检查是否已在 KG 中存在

---

## 架构总览

```
Findings (summary, sources, papers)
         ↓
AssertionGenerator (LLM) → 3 个知识断言
         ↓
EmbeddingService → 向量嵌入
         ↓
AssertionIndex (HNSW) → 相似度搜索
         ↓
KnowledgeAssertionEvaluator → quality = (新断言数/3) × 10
         ↓
QualityV2Assessor (双通道评估)
```

## 需要确认的关键问题

### 问题 1: Embedding 服务选择 (优先级: 高)

当前项目没有 embedding 服务，需要实现。有两种方案:

**方案A**: 使用 API 服务 (推荐)
- **优点**: 无需本地模型，即开即用，嵌入质量高
- **缺点**: 依赖网络，有 API 成本
- **实现**: 支持 Volcengine/OpenAI，通过环境变量配置

**方案B**: 本地模型
- **优点**: 无网络依赖，无 API 成本，隐私性好
- **缺点**: 需要下载模型 (~80MB)，推理速度较慢
- **实现**: 使用 sentence-transformers all-MiniLM-L6-v2

**方案C**: 混合模式 (最灵活)
- API 作为主要 provider，本地模型作为 fallback
- 当 API 不可用时自动切换到本地模型

**我的建议**: 方案C - 先尝试 API，失败时自动 fallback 到本地模型

### 问题 2: 向量索引技术选择 (优先级: 高)

需要存储和搜索 assertion embeddings:

**方案A**: 纯 SQLite + 线性扫描 (最简单)
- 实现: 将 embeddings 存为 BLOB，查询时全表扫描
- 性能: O(N)，1000 条内可用，1万条后明显变慢
- 代码复杂度: 低

**方案B**: SQLite + faiss-cpu (推荐)
- 实现: 使用 Facebook 的 FAISS 库做内存索引
- 性能: O(log N)，10万条也能 < 10ms
- 依赖: 需要安装 faiss-cpu

**方案C**: sqlite-vss 扩展
- 实现: SQLite 的向量相似度扩展，原生支持 HNSW
- 性能: O(log N)，性能优秀
- 依赖: 需要编译安装 sqlite-vss

**我的建议**: 方案B - faiss-cpu 成熟稳定，安装简单 (`pip install faiss-cpu`)

**请确认**: 你倾向于哪种索引方案？是否介意增加 faiss-cpu 依赖？

### 问题 3: 相似度阈值设定 (优先级: 中)

判断两个断言是否"实质相同"的阈值:

**选项A**: 固定阈值 0.82
- 基于 all-MiniLM-L6-v2 的经验值
- 优点: 简单直接
- 缺点: 不同领域可能需要不同阈值

**选项B**: 自适应阈值
- 初始 0.82，根据校准结果自动调整
- 优点: 更准确
- 缺点: 需要校准数据集

**选项C**: 多阈值分层
- 0.85+: 确定相同
- 0.75-0.85: 可能相同，用 LLM 二次确认
- < 0.75: 确定不同

**我的建议**: 先实现选项A (0.82)，后续根据实际效果调整

**请确认**: 你接受 0.82 作为初始阈值吗？

### 问题 4: 质量评估整合策略 (优先级: 高)

断言评估和原有 QualityV2 如何整合:

**方案A**: 断言评估为主，Legacy 为 fallback
- 断言评估失败时用 Legacy
- 断言给出 0.0 时直接返回
- 代码逻辑最简单

**方案B**: 智能共识 (推荐)
- 两者都高 → 用断言 (更精准)
- 两者都低 → 用较低值
- 断言 0.0 但 Legacy 高 → 用 Legacy (断言可能错了)
- 断言高但 Legacy 低 → 用断言 (可能是新知识)

**方案C**: 加权平均
- Quality = 0.6 × 断言 + 0.4 × Legacy
- 简单但可能掩盖问题

**我的建议**: 方案B - 防止断言生成失败导致的误判

**请确认**: 你同意使用智能共识策略吗？

### 问题 5: 冷启动问题处理 (优先级: 中)

第一次使用时 assertion index 是空的:

**方案A**: 什么都不做
- 第一个探索的所有断言都会被标记为"新"
- 简单但会导致初期质量分数虚高

**方案B**: 预置基础断言 (推荐)
- 预置 50-100 条 CS/AI 基础概念断言
- "Machine learning is a subset of AI"
- 避免对基础知识的误判

**方案C**: 从现有 KG 反向生成
- 遍历现有 topic summaries，生成 assertions
- 需要运行迁移脚本

**我的建议**: 方案B + 方案C - 先预置基础断言，再提供迁移脚本处理现有数据

**请确认**: 你同意预置基础断言库吗？

### 问题 6: 断言存储策略 (优先级: 中)

生成的断言是否要永久存储:

**选项A**: 只存 embeddings，不存原文
- 节省空间，但无法查看历史断言
- 调试困难

**选项B**: 存原文 + embeddings (推荐)
- 可以审计和调试
- 支持后续分析 (如统计最常见的断言)
- 占用更多存储 (每条 ~5KB)

**选项C**: 分级存储
- 高质量断言永久存储
- 低质量断言临时存储，定期清理

**我的建议**: 选项B - 全部存储，存储成本不高，调试价值大

**请确认**: 你希望存储断言原文吗？

## 详细实施计划

如果你确认以上问题，以下是详细的实施步骤:

### Phase 1: 基础设施 (2-3 天)

**Task 1.1**: 扩展配置系统
- 文件: `core/config.py`
- 新增: `EmbeddingConfig` dataclass
- 配置项: provider, model, dimension, threshold, fallback_chain

**Task 1.2**: 实现 EmbeddingService
- 文件: `core/embedding_service.py`
- 功能: 多 provider 支持 (API + 本地)
- 特性: 自动 fallback, LRU cache

**Task 1.3**: 实现 AssertionIndex
- 文件: `core/assertion_index.py`
- 技术: SQLite + FAISS (或纯 SQLite)
- 功能: insert, search_similar, max_similarity

**Task 1.4**: 实现 AssertionGenerator
- 文件: `core/assertion_generator.py`
- 功能: LLM prompt, 断言解析, 质量过滤
- 过滤: 长度检查, 禁用前缀, topic 重复检测

### Phase 2: 核心逻辑 (2-3 天)

**Task 2.1**: 实现 KnowledgeAssertionEvaluator
- 文件: `core/knowledge_assertion_evaluator.py`
- 功能: assess_quality, is_assertion_new, index_assertion
- 整合: 检查 assertion index + KG summaries

**Task 2.2**: 修改 QualityV2Assessor
- 文件: `core/quality_v2.py`
- 新增: assertion_evaluator 初始化
- 修改: assess_quality 使用双通道评估
- 实现: 智能共识聚合逻辑

**Task 2.3**: 添加冷启动支持
- 文件: `scripts/seed_baseline_assertions.py`
- 预置: 50-100 条基础 CS/AI 断言

### Phase 3: 测试验证 (1-2 天)

**Task 3.1**: 单元测试
- 文件: `tests/core/test_knowledge_assertion.py`
- 覆盖: assertion 生成, 相似度检测, 质量评估

**Task 3.2**: 集成测试
- 文件: `tests/integration/test_quality_v2_integration.py`
- 场景: 高质 findings → 高分, 低质 findings → 低分

**Task 3.3**: 校准测试
- 文件: `tests/calibration/test_similarity_threshold.py`
- 目标: 确定最优相似度阈值

### Phase 4: 部署准备 (1 天)

**Task 4.1**: 迁移脚本
- 文件: `scripts/migrate_to_assertion_quality.py`
- 功能: 从现有 KG 生成 assertions

**Task 4.2**: 配置更新
- 文件: `config.json` 示例更新
- 文档: README 更新
