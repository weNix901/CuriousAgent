## 一、现状审视：当前双向交互机制全图

### 1.1 现有交互路径（实际实现）

```

│                                                                              │

│       │                                                                      │

│       │         └──► 读取 state.json → 同步到 memory/curious/              │

│       │                                                                      │

                                     │ 唯一桥接：trigger_explore.sh

┌──────────────────────────────────────────────────────────────────────────────┐
│                            Curious Agent                                    │
│                                                                              │
│  curiosity_queue (state.json)                                               │

│       ├──► 探索引擎 → findings → state.json                                 │
│       │                                                                      │
│       ├──► AgentBehaviorWriter (quality >= 7.0)                             │
│       │         ├──► curious-agent-behaviors.md                             │
│       │         └──► memory/curious/{topic}.md (#behavior-tag)             │
│       │                                                                      │
│       └──► sync_discoveries.py (由 R1D3 心跳拉取)                          │

└──────────────────────────────────────────────────────────────────────────────┘
```

### 1.2 当前机制的核心问题

| 问题 | 描述 | 影响 |
|------|------|------|
| **写入路径重复且不兼容** | `AgentBehaviorWriter._sync_to_memory()` 和 `sync_discoveries.py` 写入同一目录但 schema 互不兼容 | memory/curious/ 里同时存在 `[behavior]` 和 `[curious]` 两种 header，无法统一检索 |
| **R1D3 学习需求无结构化写入** | R1D3 说"我想了解 X"，只能通过手动 shell 命令触发 inject | 无法自动化，无法持久化意图 |
| **结果回流靠轮询** | R1D3 等心跳同步（最长 30min 延迟），无主动通知机制 | 实时性差，资源浪费 |
| **无交叉校验** | R1D3 的理解和 Curious Agent 的发现没有关联记录 | 无法判断谁更准确 |
| **好奇 topic ≠ 学习需求** | "Agent 自我认知"（R1D3 需求粒度）和 "Agent self-awareness"（Curious topic 粒度）可能不对齐 | 注入后探索方向可能偏 |
| **shared 标记不完整** | sync_discoveries.py 写 shared:false，但 AgentBehaviorWriter 写的是 behavior 格式，不走索引 | 部分文件永远不被 R1D3 感知 |

### 1.3 当前各写入路径详情

| 写入方 | 触发条件 | 目标文件 | Header格式 | shared 标记 |
|--------|---------|---------|-----------|-------------|
| `AgentBehaviorWriter._sync_to_memory()` | quality >= 7.0 | `memory/curious/{topic}.md` | `# [behavior]` | 无 |
| `sync_discoveries.py` | R1D3 心跳轮询 | `memory/curious/{topic}.md` + `curious-discoveries.md` | `# [curious]` | `shared: false` |
| `trigger_explore.sh` | R1D3 手动调用 | (不写文件，只调 API) | N/A | N/A |

## 二、双体协作读写契约设计

### 2.1 核心原则

> **契约优先于实现。读写契约定义系统间的承诺，不定义内部细节。**

1. **单向写入，双向读取**：每个数据流方向由单一系统负责写入，多系统可以读取
2. **格式即契约**：文件 schema 是系统间唯一的协议，不依赖内部状态
3. **接口幂等性**：重复写入同一数据应 idempotent（覆盖而非重复）
4. **可演进性**：schema 版本化，旧版本可读，新版本优先

### 2.2 数据流方向定义

```
┌──────────────────────────────────────────────────────────────────────────────┐
│                              数据流方向定义                                   │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  R1D3 ──────────写入─────────► shared_knowledge/r1d3/learning_needs/       │
│       │                               ↓                                      │
│       │                          (结构化学习需求)                             │
│       │                               ↓                                      │
│       │                    Curious Agent 消费                                │
│       │                               ↓                                      │
│       │                    curiosity_queue (内部)                            │
│       │                               ↓                                      │
│       │                       探索完成 → findings                             │
│       │                               ↓                                      │
│       │                    shared_knowledge/curious/                         │
│       │                               ↓                                      │
│       │                    R1D3 读取（心跳/主动）                             │
│       │                               ↓                                      │
│       │                    memory_search 可检索                              │
│       │                                                                       │
│  写入方（绝对权威）          消费方（只读不写）                                │
└──────────────────────────────────────────────────────────────────────────────┘
```

## 三、R1D3 学习需求写入机制设计

### 3.1 触发场景分类

| 场景 | R1D3 行为 | 产生的 learning_need 类型 |
|------|----------|-------------------------|
| 用户问了一个 R1D3 不确定的问题 | 诚实回答 + 记录需求 | `question_derived` |
| R1D3 主动感知到知识缺口 | 主动记录需求 | `self_perceived` |
| R1D3 触发好奇探索（"让我好奇一下"） | 触发时同步写入需求 | `curiosity_driven` |
| R1D3 读取到新的 curious_finding | 产生新的疑问 | `discovery_derived` |

### 3.2 写入 API 设计

**内部函数**：`write_learning_need(topic, need_type, context)`

```python
def write_learning_need(
    topic: str,
    need_type: str,          # question_derived | self_perceived | curiosity_driven | discovery_derived
    context: dict
) -> str:
    """
    写入 R1D3 学习需求到 shared_knowledge/r1d3/learning_needs/
    
    Args:
        topic: 学习主题
        need_type: 需求类型
        context: {
            "r1d3_description": str,      # R1D3 用自己的话描述
            "trigger_scenario": str,       # 触发场景描述
            "confidence_before": float,    # 当前置信度 0-1
            "existing_knowledge": str,     # 已有理解摘要
            "granularity_hint": str,       # "broad" | "specific"
            "curious_topic_hint": str,    # 给 Curious Agent 的 topic 建议
        }
    
    Returns:
        filename: 写入的文件路径
    """
```

### 3.3 与 trigger_explore.sh 的关系

**当前问题**：`trigger_explore.sh` 只调 API，不写文件，无法持久化 R1D3 的意图。

**演进方案**：

```
R1D3 调用 trigger_explore.sh
         │
         ├──► 写入 shared_knowledge/r1d3/learning_needs/{topic}.md
         │         (持久化 R1D3 的学习需求)
         │
         └──► POST /api/curious/inject
                   (通知 Curious Agent 立即探索)
```

**演进后 `trigger_explore.sh` 行为**：
1. 写入 learning_need 文件（持久化）
2. 调用 `/api/curious/inject`（触发探索）
3. learning_need 的 `curious_topic_hint` 作为 inject 的 `topic` 参数

### 3.4 Curious Agent 对 learning_needs 的消费

**新增 API**：`GET /api/curious/learning_needs`

```json
{
  "pending": [
    {
      "filename": "2026-03-25-agent-self-awareness.md",
      "topic": "Agent self-awareness",
      "need_type": "curiosity_driven",
      "confidence_before": 0.4,
      "created_at": "2026-03-25T14:00:00Z",
      "granularity_hint": "specific",
      "curious_topic_hint": "agent_self_awareness_mechanisms"
    }
  ],
  "completed": [...],
  "conflict": [...]
}
```

**内部逻辑**：
1. 每轮探索前检查 `pending` 列表
2. 如果有 pending → 优先注入到 curiosity_queue（带 `r1d3_learning_need_ref`）
3. 探索完成 → 更新 `curious_status` 和 `curious_finding_ref`
4. 如果 R1D3 的理解和 Curious 发现有重大差异 → `curious_status: conflict`

## 四、Curious Agent 侧改造

### 4.1 新增 API 端点

| 端点 | 方法 | 用途 |
|------|------|------|
| `/api/curious/learning_needs` | GET | 获取所有 pending/completed/conflict 的 learning_needs |
| `/api/curious/learning_needs/<id>` | PATCH | 更新某个 learning_need 的状态 |
| `/api/curious/learning_needs/subscribe` | POST | R1D3 订阅 learning_needs 变更通知 |

### 4.2 行为写入路径统一

**现状**：
- `AgentBehaviorWriter._sync_to_memory()` → 写 `memory/curious/{topic}.md` (#behavior)
- `sync_discoveries.py` → 写 `shared_knowledge/curious/{topic}.md` (#curious)

**演进后**（v0.2.4）：
- Curious Agent **统一写入** `shared_knowledge/curious/{date}-{topic}.md`
- `shared_knowledge/curious/` 替代 `memory/curious/` 成为权威目录
- `memory/curious/` 保持只读兼容，不删除已有文件

**AgentBehaviorWriter 改动**：
- `_sync_to_memory()` 目标改为 `shared_knowledge/curious/`
- 写完后**同时**更新 `shared_knowledge/curious/index.md`（统一索引）

### 4.3 R1D3 inject 优先机制

**背景**：R1D3 注入的话题应该有更高优先级或即时触发。

**实现**：
- inject API 检测 `source == "r1d3"`
- 如果 `config.exploration.injection_priority.trigger_immediate == True`：
  - 立即将 topic 移到 curiosity_queue 顶部
  - 异步触发一轮探索（不等待完成）
- 如果 `boost_score > 0`：应用分数加成

## 五、演进路线

### Phase 0: 共享知识层基础设施（不影响现有功能）

1. 新建 `shared_knowledge/{r1d3,curious,cross_validation}/` 目录
2. 实现 `write_learning_need()` 函数
3. 实现 Curious Agent 的 `/api/curious/learning_needs` GET 端点
4. **不删除** 任何现有文件，保持兼容

**验收标准**：新旧文件可以共存，R1D3 可以读写 shared_knowledge/

---

### Phase 1: R1D3 学习需求 → Curious Agent 自动化流

1. R1D3 的 `trigger_explore.sh` 增加写入 `shared_knowledge/r1d3/learning_needs/`
2. Curious Agent 的调度循环增加消费 `learning_needs` 的逻辑
3. 探索完成后填写 `curious_finding_ref` 回流

**验收标准**：R1D3 说"让我好奇一下 X"，3 分钟内 curiosity_queue 有记录

---

### Phase 2: Curious Agent 结果 → R1D3 主动通知

1. Curious Agent 探索完成且 quality >= 7.0 → **主动通知** R1D3
2. 方式：写入一个「待通知」队列文件（`shared_knowledge/notify_queue/{id}.json`）
3. R1D3 心跳检测到这个文件 → 读取 → 告知用户 → 删除文件

**验收标准**：高质量发现从完成到用户感知 < 2 分钟（不再是 30 分钟心跳轮询）

---

### Phase 3: 交叉校验机制

1. R1D3 读取 curious_finding 后，写入 `cross_validation/{topic}.md`
2. Curious Agent 下次探索同 topic 时读取交叉校验记录
3. 如果 `status == conflict`，降低该 topic 的探索优先级

**验收标准**：同一 topic 的 R1D3 理解和 Curious 发现可以对比

---

### Phase 4: 旧格式迁移（可选）

1. 扫描 `memory/curious/` 中的所有文件
2. 转换为 `shared_knowledge/curious/` 格式
3. 更新 `curious-discoveries.md` 索引指向新目录
4. 旧目录标记为 deprecated（只读）

**验收标准**：`memory/curious/` 可安全归档
