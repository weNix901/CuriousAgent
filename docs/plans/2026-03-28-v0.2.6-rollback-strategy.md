# v0.2.6 Rollback Strategy

## Overview

This document outlines the rollback strategy for Curious Agent v0.2.6 (Dream Agent) in case critical issues are discovered after deployment.

## Emergency Rollback

### Immediate Rollback Command

If serious issues are found after deployment, execute immediate rollback:

```bash
# Emergency rollback to v0.2.5
git checkout feature/v0.2.5-stable

# Or use tag:
git checkout v0.2.5

# Restart the service
python curious_agent.py --daemon
```

### Data Compatibility

- **Backward Compatible**: v0.2.6 adds fields that v0.2.5 ignores
- **No Data Migration**: No migration needed for rollback
- **Ignored by v0.2.5**:
  - `dream_insights/` directory
  - `dream_topic_inbox.json`
  - New topic fields (status, dreamed_at, confidence_*)

## Feature Toggle

### Configuration

Add to `config.json` to disable v0.2.6 features:

```json
{
  "v026_features": {
    "enable_three_agents": false,
    "spider_enabled": true,
    "dreamer_enabled": true,
    "pruner_enabled": true,
    "pruner_interval_minutes": 240
  }
}
```

### Usage

1. Set `enable_three_agents: false` to disable three-agent mode
2. Service will automatically fall back to legacy daemon mode
3. No restart required (config checked on each cycle)

## Rollback Checklist

### Before Rollback

- [ ] Confirm issue is critical (data loss, system crash, deadlock)
- [ ] Document issue for post-mortem
- [ ] Notify team members
- [ ] Check if feature toggle can resolve issue (preferred)

### During Rollback

- [ ] Stop current service: `Ctrl+C` or kill signal
- [ ] Verify no data corruption in KG
- [ ] Checkout v0.2.5: `git checkout v0.2.5`
- [ ] Restart service: `python curious_agent.py --daemon`
- [ ] Verify service starts without errors

### After Rollback

- [ ] Monitor logs for errors
- [ ] Verify exploration continues normally
- [ ] Document rollback completion time
- [ ] Schedule post-mortem meeting
- [ ] Create issue ticket for v0.2.6 fix

## Known Issues & Workarounds

### Issue 1: NodeLockRegistry Not Used by Legacy add_knowledge()

**Symptom**: Concurrency stress test shows `kg` module functions don't use NodeLockRegistry

**Impact**: Low - legacy functions work but without optimal locking

**Workaround**: 
- Use feature toggle to disable three-agent mode
- Continue with legacy single-thread mode

**Fix**: Update legacy functions to use NodeLockRegistry (planned for v0.2.7)

### Issue 2: DreamAgent LLM Cost

**Symptom**: High LLM API costs from DreamAgent

**Impact**: Medium - cost increase

**Workaround**:
- Adjust config: `dreamer_enabled: false`
- Or increase quality threshold in DreamAgent

### Issue 3: KG Growth

**Symptom**: KG growing too fast

**Impact**: Low - storage concerns

**Workaround**:
- Decrease pruner interval: `pruner_interval_minutes: 120`
- Or adjust SleepPruner dormancy criteria
